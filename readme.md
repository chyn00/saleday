# SaleDay

사용자가 몰리는 상황에서도 **주문과 재고 데이터의 일관성**을 지키도록 설계한 커머스 백엔드 프로젝트입니다.  
목표는 단순 성능 최적화가 아니라, **빠른 응답과 데이터 정합성**을 함께 달성하는 것입니다.

**기간:** 2025.03 ~ (지속적 설계·검증 및 고도화)

---

## 시작 전 안내

- 일부 설정은 `local` 프로파일에 의존하므로 바로 실행이 어려울 수 있습니다.
- Kafka / Redis / MySQL 인프라가 준비되지 않으면 일부 기능은 동작하지 않습니다.

---

## 실행 방법 (Docker Compose)

프로젝트 루트에서 실행:

```bash
docker compose up -d --build
```

확인:

```bash
docker ps
```

기본 포트:

- API: `http://localhost:9999`
- Consumer: `localhost:9998`
- Kafka: `localhost:9092`
- Redis: `localhost:6379`
- MySQL: `localhost:3306`

중지:

```bash
docker compose down
```

## 핵심 설계 관점

### 1) 재고 하한 제어

재고는 어떤 경우에도 0 미만으로 내려가지 않도록 제어했습니다.

- Redis `DECR`로 재고 선차감
- 감소 결과 `< 0`이면 즉시 보정 후 실패 처리
- DB 반영은 이후 비동기 단계에서 수행

Redis는 영속 저장소가 아니라 **빠른 판단 계층**으로 제한해 사용했습니다.

### 2) 동기/비동기 경계 설정

모든 처리를 동기 트랜잭션으로 묶으면, 고트래픽에서 DB 경합과 대기 시간이 빠르게 증가합니다.

- **동기 구간**: 주문 가능 여부(재고 감소 가능 여부) 즉시 판단
- **비동기 구간**: 주문 커밋 이후 DB 재고 반영

이렇게 경계를 나눠, 사용자 응답 지연과 후속 반영 책임을 분리했습니다.

### 3) Outbox + 재처리 구조

비동기 구조에서 핵심은 "실패 이후에도 최종 상태를 맞출 수 있는가"입니다.

- 주문 트랜잭션 내부에서 Outbox 기록
- 상태 관리: `INIT / SUCCESS / FAILED`
- `FAILED` 이벤트 재전송 스케줄링
- Consumer 멱등 처리

즉시 일치가 아닌 구간도, 재처리를 통해 **최종 일관성**으로 수렴하도록 설계했습니다.

### 4) 락 전략 비교와 선택 근거

이 프로젝트에서 락 전략은 "무엇이 가장 엄격한가"보다, **트래픽 대비 병목·운영 복잡도·정합성 수준**을 함께 비교해 선택했습니다.

- **비관적 락(DB Lock)**
  - 장점: 충돌 제어가 명확함
  - 단점: 동시 요청이 몰릴 때 대기/경합이 커지고 DB 병목으로 이어지기 쉬움
  - 판단: 고트래픽 주문 경로의 1차 선택지로는 비효율적

- **낙관적 락(Version/조건 갱신)**
  - 장점: 락 점유 없이 충돌 감지 가능
  - 단점: 충돌 빈도가 높아지면 재시도 비용이 커지고 실패율이 올라갈 수 있음
  - 판단: 충돌이 상대적으로 낮은 도메인에는 유효하지만, 집중 트래픽 재고 차감에는 불리할 수 있음

- **Redis 분산락**
  - 장점: 애플리케이션 레벨에서 동시성 제어 가능
  - 단점: 락 획득/해제 관리, 타임아웃, 장애 시 처리 정책 등 운영 복잡도가 증가
  - 판단: 정합성 제어엔 유효하지만, 초당 다건 차감 경로에서는 비용 대비 성능 손실을 신중히 봐야 하기 때문에 선택하지 않음

- **현재 선택: Redis 원자 연산 + 비동기 반영**
  - 장점: 빠른 응답과 재고 하한 제어를 동시에 달성하기 유리
  - 보완: Outbox/재처리/멱등 소비로 최종 상태 일치 보장

추가로, "즉시 조회 일관성" 요구가 강한 도메인이라면 Read Projection을 분리해 설계하거나, DB 기반으로 읽기 모델을 별도 운영하는 방식도 현실적인 대안으로 보고 있습니다.

## 아키텍처

- 멀티 모듈 구조
  - `api`: 요청 처리, Outbox 기록, Kafka Producer
  - `order`: 주문 / 재고 도메인
  - `item`: 상품 / 리뷰
  - `discount`: 할인 정책
  - `consumer`: Kafka Consumer
  - `pay`: 결제 연동
  - `common`: 공통 설정 및 Outbox 모델
  - `message`: 이벤트 계약
- Redis 기반 재고 감소 제어
- Outbox 기반 이벤트 발행
- Kafka Consumer를 통한 DB 반영

## 테스트 전략

테스트의 목적은 "코드 실행 확인"이 아니라,  
**실패 상황에서도 데이터 일관성을 설명 가능한 수준으로 검증**하는 데 두었습니다.

### 1) 단위 테스트 우선: 핵심 분기 고정

외부 의존성(Kafka, Redis, 외부 API)은 mock으로 격리하고, 도메인 분기와 상태 전이를 빠르게 검증했습니다.

- 재고 부족 시 즉시 실패 + 보상 로직
- 주문 실패 시 재고 롤백(보상 트랜잭션)
- Kafka 발행 성공/실패에 따른 Outbox 상태 전이
- Consumer 처리 시 멱등/락 분기

핵심은 성공 케이스보다 **실패 케이스를 먼저 고정**해 회귀를 빠르게 잡는 것입니다.

### 2) 통합 테스트는 경계 중심

전체를 통합 테스트로 덮지 않고, 장애가 실제로 발생하는 경계만 선택 검증했습니다.

- 메시징 경계: Producer → Kafka → Consumer
- 데이터 경계: DB 반영과 Outbox 상태 일치
- 동시성 경계: 재고 감소 처리와 주문 흐름 연결

### 3) 비동기 테스트 일관성 확보

비동기 테스트는 실행 타이밍에 따라 결과가 흔들릴 수 있어, 테스트 환경에서는 executor를 동기 방식으로 주입해 검증 흐름을 고정했습니다.

- `CompletableFuture` 성공/실패 분기 재현 가능성 확보
- 스레드풀 튜닝과 처리량 비교는 성능 테스트로 분리

### 4) 성능 테스트 + 정합성 검증 동시 수행

부하 테스트는 응답 시간만 보는 용도가 아니라, 최종 데이터 일치 여부를 함께 검증하기 위해 수행했습니다.

```text
초기재고 - 주문수량합 = DB재고 = Redis재고
```

- `order_item` 수량 합 조회
- `item_stock` 조회
- Redis 재고 키 조회
- Outbox 상태(`INIT/SUCCESS/FAILED`) 점검

즉, 이 프로젝트의 테스트는  
**기능 검증 + 실패 복구 검증 + 데이터 정합성 검증**을 한 세트로 다룹니다.

- 1,000 TPS

## AI Agent 활용

AI Agent는 설계를 대체하는 도구가 아니라, 검증과 반복 작업 속도를 높이는 보조 도구로 사용했습니다.

- 테스트 코드 작성 가속
- 부하 테스트 반복 실행 자동화
- 정합성 검증 리포트 정리
- 대안 비교 시 논점 정리 보조

동기/비동기 경계 설정, 락 전략 선택, 재처리 정책, 최종 구조 결정은 개발자가 직접 수행했습니다.

## 프로젝트를 통해 얻은 것

- 정합성과 성능은 선택 문제가 아니라 **경계 설계 문제**라는 점
- 고트래픽에서는 동기/비동기 분리와 실패 복구 설계가 핵심이라는 점
- 성능 수치와 데이터 일치 검증을 반드시 함께 봐야 한다는 점
- AI Agent를 활용하더라도 설계 책임은 개발자가 가져가야 한다는 점
